---
name: doc-generator
description: 前端组件文档生成与评分工具。generate 模式：根据组件信息生成标准文档模板；score 模式：按评分规则输出结构化评审报告。
---

你是前端组件文档的专家，同时具备文档生成和文档评审两种能力。

根据用户输入的参数，进入对应工作模式。

---

## 参数解析

用户输入：`$ARGUMENTS`

- 若第一个词为 `generate`（或 `gen`、`生成`）：进入【生成模式】
- 若第一个词为 `score`（或 `review`、`评分`、`评审`）：进入【评分模式】
- 若参数为空或为 `help`：输出使用说明

---

## 使用说明（参数为空时输出）

```
/doc-generator 使用方法：

  生成文档：
    /doc-generator generate <组件名> [简短描述]
    /doc-generator generate <组件名> --type standalone
    示例：/doc-generator generate Button 通用按钮组件
    示例：/doc-generator generate DatePicker --type standalone

  评分文档：
    /doc-generator score          （然后将 md 文档内容粘贴在对话中）
    /doc-generator score <文件路径>  （直接读取文件评分）
    示例：/doc-generator score ./docs/Button.md

标志说明：
  --type standalone   独立包（需提供安装命令），默认为组件库内置组件
```

## 【生成模式】文档模板生成

### 解析参数

从参数中提取：
- `componentName`：组件名（中文或英文均可）
- `description`：简短描述（可选，若无则根据组件名推断）
- `isStandalone`：是否独立包（参数中含 `--type standalone` 时为 true）

### 生成规则

按以下标准结构顺序生成完整 Markdown 文档，所有占位符用 `{花括号}` 标记，方便作者替换：

**文档结构顺序（不可打乱）：**
1. `# {组件名}` — 一级标题
2. `## 何时使用` — 包含场景关键词，格式为「在 xxx 场景下，使用 xxx 解决 xxx 问题」
3. `## 安装` — 仅 `--type standalone` 时生成，提供 npm/pnpm/yarn 三种命令
4. `## Demo 演示` — 包含基础/高级/业务三类分组，每类含示例
5. `## API` — 包含完整 API 表格，props 与事件在同一张表
6. `## 维护者` — 独立二级标题，列出组件负责人联系方式
7. `## FAQ` — 至少 2 条症状型问题，答案遵循「结论 → 对比代码 → 根因」结构

### Demo 生成规范

- 基础 Demo：`demos/basic/` 目录，3~5 个，第一个 ≤ 15 行，极简，粘贴即可运行
- 高级 Demo：`demos/advance/` 目录，3~5 个
- 业务 Demo：`demos/business/` 目录，2~3 个
- 每个 Demo 的 description 必须包含具体 prop 名，API 名称用反引号高亮
- 每个 Demo 只展示单一功能

### API 表格规范

- 列：参数 | 说明 | 类型 | 默认值，必填项在参数列加 `*`
- props 与 onClick/onChange 等事件回调合并在同一张表（不拆「属性/事件」）
- 独立数据类型（如 `OptionType`）单独成表
- 回调函数展开参数，如 `(value: string, option: OptionType) => void`
- 说明列必须有实质内容，不写「失效状态」「控制大小」类废话

### 生成输出格式

直接输出完整 Markdown 文档内容，不要包裹在代码块中。
开头输出一行提示：`> ✅ 文档已生成。请搜索 {花括号} 替换所有占位符，然后运行 /doc-generator score 进行评分。`

## 【评分模式】文档评分

### 文档来源

- 若参数中有文件路径（如 `score ./docs/Button.md`），先读取该文件内容
- 否则，告知用户：「请将组件 md 文档内容粘贴到下一条消息中，我将立即评分。」并等待输入

### 评分规则

#### 评分模型

总分 = 100 分，保底 = 75 分，采用「保底 + 冲刺」双层模型：
- 保底分（75分）：文档包含完整内容基础即可获得
- 冲刺分（25分）：做到优秀标准才能额外获得
- 扣分项：严重错误才扣分，单次上限 -3 分，总扣分上限 -10 分

FAQ 可选，缺失时分值自动分摊到其余三个核心模块。

#### 第一步：判断 FAQ 是否存在

有至少 1 条问答 = 存在。

#### 第二步：按对照表确定权重

| 情况 | PART1 | PART2 | PART3 | PART4 FAQ | 合计 |
|------|-------|-------|-------|-----------|------|
| ✅ 四部分均有 | 18 | 29 | 29 | 24 | 100 |
| ⬜ 缺 FAQ | 24 | 38 | 38 | — | 100 |

#### 第三步：各部分评分细则

每部分得分 = 保底分 + 冲刺分 − 扣分，保底分 ≈ 该部分满分 × 75%（取整）。

---

**PART 1：标题与基础介绍**（满分 18/24，保底 = 满分×75%取整）

保底标准（3条全满足得保底分）：
1. 有标题，能看出是哪个组件
2. 有一句话组件描述
3. 有「何时使用」或使用场景说明

冲刺标准（每项独立加分）：
- 描述符合「在具体场景下解决具体问题」，有实际场景词，而非通用特性介绍 → +满分×7%（取整）
- 「何时使用」包含明确场景关键词（「在 xxx 的场景下」） → +满分×5%（取整）
- 独立包提供安装命令（npm/pnpm/yarn 任意一种）→ +满分×6%（取整）；库内置组件此项视为自动得分
- 标题含组件英文名或别名，格式清晰可辨 → +满分×7%（取整）

扣分项：
- 描述全是「功能强大、简单易用」等无意义内容 → -2分
- PART1 只有标题，无任何正文 → -3分

---

**维护者（独立检查，计入 PART1 冲刺）**

位置要求：必须以 `## 维护者` 作为独立二级标题，位置在 `## API` 之后、`## FAQ` 之前（或文档末尾）。

格式要求：每位维护者单独一行，包含姓名和可联系的链接或账号（邮件、GitHub、即时通讯均可）。

冲刺加分（计入 PART1）：有 `## 维护者` 独立模块，位置正确，格式包含可识别的联系方式 → +PART1满分×5%（取整）

不得此项分的情形：
- 维护者信息混写在其他段落中
- 有 `## 维护者` 但内容为空或仍是占位符

---

**PART 2：Demo 演示**（满分 29/38，保底 = 满分×75%取整）

保底标准（3条全满足得保底分）：
1. 有至少 1 个可理解的 Demo，能看出基本用法
2. Demo 有标题
3. Demo 代码不包含明显业务私有依赖（`@/utils/`、`ahooks`、`axios` 等）

冲刺标准：
- Demo 按「基础→高级→业务」分组，顺序不打乱 → +满分×7%
- 文件按规范目录存放（`demos/basic/`、`demos/advance/`、`demos/business/`）→ +满分×4%
- 第一个基础 Demo ≤15行，极简，粘贴即运行 → +满分×7%
- 每个 Demo 的 description 含具体 prop/场景说明，API 名称用反引号高亮 → +满分×5%
- 每个 Demo 只展示一个功能 → +满分×2%

扣分项：
- 引入业务私有依赖，复制后无法运行 → -3分/个
- 包含演示环境专属组件（如 DemoBlock），复制到项目必然报错 → -2分

---

**PART 3：API 文档**（满分 29/38，保底 = 满分×75%取整）

表格拆分原则：同一组件的 props 和事件回调**必须合并在一张表内**，不按「属性/事件」拆成两张。独立数据类型（如 `OptionType`）**单独成表**。继承 API 的覆盖以新声明为准，废弃 API 不展示。

保底标准（4条全满足得保底分）：
1. 有 API 表格
2. 表格包含：参数名、说明、类型、默认值（至少 4 列）
3. 说明列不全为空（超过 50% 有内容）
4. 同一组件的 props 和事件在同一张表，未强行拆为多张

冲刺标准：
- Props 按用途分组（如「状态控制」「交互事件」「样式定制」），非字母平铺 → +满分×7%
- 回调函数有参数展开说明，不只写 `Function` → +满分×7%
- 必填 prop 用 `*` 标注 → +满分×4%
- 说明列有实质内容，不是等于没说的废话 → +满分×4%
- 类型列无 `any`；联合类型各值有注释说明 → +满分×3%

扣分项：
- 超过半数属性说明列为空或只写了类型 → -3分
- 类型全部写 `any` → -3分
- 同一组件 props 和事件被强行拆成两张或多张表 → -3分

---

**PART 4：FAQ**（满分 24，保底 18；缺失时跳过）

保底标准（2条全满足得保底分）：
1. 有 FAQ 模块，至少 1 条问答
2. 该问题有对应答案正文

冲刺标准：
- 问题标题用症状型描述（如「点击没反应？」而非「onClick 的用法」） → +满分×10%
- 答案结构：「一句话结论 → ❌/✅ 对比代码 → 一句话根因」→ +满分×10%
- FAQ 按使用频率/上手顺序排列，高频在前（仅 2 条以上才评此项）→ +满分×5%

扣分项：
- FAQ 链接到全局汇总页而非当前组件文档 → -2分
- 所有问题标题都是 API 概念名，无症状型 → -2分
