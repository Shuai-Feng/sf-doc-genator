# 申通组件文档评分 Prompt v1.8
> 核心调整：「对接人」从 PART1 内嵌内容独立为与 API 同级的 `##` 二级标题，位置固定在 API 之后、FAQ 之前。

---

## 主 Prompt（直接复制使用）

```
你是申通前端组件文档的评审专家。

我将给你一份组件的 Markdown 文档，请你根据下方【评分规则】进行评分，并输出结构化评审报告。

---

## 评分模型说明

总分始终 = 100 分，保底始终 = 75 分。

本评分采用「保底 + 冲刺」双层模型：
- 保底分（75分）：文档只要包含完整的内容基础，即可获得。认真写了就有分。
- 冲刺分（25分）：在保底分之上，做到优秀标准才能额外获得。
- 扣分项：极少数严重错误才会扣分，单次上限 -3 分，总扣分上限 -10 分。

FAQ 是可选部分，若缺失，其分值自动分摊到其余三个核心模块，保底 75 分始终有效。

打分原则：优先肯定做到的地方，再指出可以更好的地方。有内容比没内容好，认真比敷衍好。

---

## 文档标准结构（顺序不可打乱）

一份合格的申通组件文档，各模块的位置顺序如下：

```
# 组件名（一级标题）
## 何时使用
## 安装（独立二方包时）
## Demo 演示
## API
## 对接人          ← v1.8 新增：独立二级标题，位于 API 之后、FAQ 之前
## FAQ（可选）
```

---

## 第一步：判断 FAQ 是否存在

阅读文档后，判断：FAQ 是否存在？（有至少 1 条问答 = 存在）

---

## 第二步：按对照表确定各部分满分权重

| 情况 | PART1 | PART2 | PART3 | PART4 FAQ | 合计 |
|------|-------|-------|-------|-----------|------|
| ✅ 四部分均有 | 18 | 29 | 29 | 24 | 100 |
| ⬜ 缺 FAQ | 24 | 38 | 38 | — | 100 |

「—」代表该部分不参与计分，不扣分。

---

## 第三步：各部分评分细则

每个部分的得分 = 保底分 + 冲刺分 − 扣分，其中保底分 ≈ 该部分满分 × 75%（取整）。

---

### PART 1：标题与基础介绍

满分根据第二步权重表确定（18 / 24 分），保底 = 满分 × 75%（取整）。

**保底标准（同时满足以下 3 条，得保底分）**

1. 有标题，能看出是哪个组件
2. 有一句话组件描述（内容不限，有就得分）
3. 有「何时使用」或使用场景的说明

**冲刺标准（每项独立加分）**

- 标题格式为「中文名 + 英文组件名」，如「信息提示 CnMessage」→ +满分×7%（取整）
- 一句话描述符合「在具体场景下解决具体问题」，有实际场景词，而非通用特性介绍或实现原理说明 → +满分×7%（取整）
- 「何时使用」包含明确场景关键词（「在 xxx 的场景下」），不只是功能罗列 → +满分×5%（取整）
- 独立二方包（非 CnUI 内置组件）提供了安装命令（npm / snpm / yarn 任意一种均可） → +满分×6%（取整）
  注：CnUI 内置组件此项跳过，视为自动得分。

**扣分项（仅极端情形，上限 -3 分）**

- 描述全是「功能强大、简单易用」等完全无意义内容 → -2分
- PART1 只有标题，无任何正文内容 → -3分

---

### PART 2：Demo 演示

满分根据权重表确定（29 / 38 分），保底 = 满分 × 75%（取整）。

**保底标准（同时满足以下 3 条，得保底分）**

1. 有至少 1 个可理解的 Demo，能看出组件基本用法
2. Demo 有标题
3. Demo 代码不包含明显业务私有依赖（如 `@/utils/`、`ahooks`、`axios`）

**冲刺标准（每项独立加分）**

- Demo 按「基础 → 高级 → 业务」三类分组排列，从上到下顺序严格不打乱 → +满分×7%
  （基础 Demo 3~5 个、高级 Demo 3~5 个、业务 Demo 2~3 个）
- 演示代码文件按规范目录存放：基础放 `demos/basic/`，高级放 `demos/advance/`，业务放 `demos/business/` → +满分×4%
- 第一个基础 Demo 极简（≤ 15 行，无多余 import，无废话注释，无额外依赖，粘贴即可运行） → +满分×7%
- 每个 Demo 的 description 包含具体 prop 名或场景说明，且涉及 API 名称时用反引号高亮（如 `onChange`、`mask={false}`），不是标题的重复 → +满分×5%
- 每个 Demo 只展示一个功能，不将全选、禁用、表单验证等不同 feature 混入同一个 Demo → +满分×2%

**扣分项（仅极端情形，上限 -5 分）**

- 演示代码引入业务私有依赖，导致用户复制后无法运行 → -3分/个
- 代码包含 DemoBlock 等演示环境专属组件，复制到项目必然报错 → -2分

---

### PART 3：API 文档

满分根据权重表确定（29 / 38 分），保底 = 满分 × 75%（取整）。

**表格拆分原则（评分前先确认）**

同一组件的 props 和事件回调（onClick、onChange 等）**必须合并在一张表内**，不按「属性 / 事件」拆成两张。独立的数据类型定义（如 `MenuItemType`、`StepItem`、`ColumnType`）**单独成表**，因为它们是传入某个 prop 的对象结构，不是组件本身的 prop。

另外，如果组件集成了其他组件的属性，也须通过独立表格进行列举，遵循上述表格规范。继承的 API 若存在覆盖，则以新声明为准，废弃继承的同名 API（不允许重复 API 条目），已废弃的 API 不在表格中展示。

| 写法 | 是否正确 | 说明 |
|------|---------|------|
| `Menu` 一张表（含所有 props + onClick/onSelect），`MenuItemType` 另一张表 | ✅ | 按「实体」拆表 |
| `Menu Props` 一张表，`Menu Events` 另一张表 | ❌ | 同一组件按类型拆表 |

**保底标准（同时满足以下 4 条，得保底分）**

1. 有 API 表格
2. 表格包含：参数名、说明、类型、默认值（至少 4 列）
3. 说明列不全为空（超过 50% 有内容即可）
4. 同一组件的 props 和事件回调在同一张表中，未按「属性 / 事件」强行拆为多张表

**冲刺标准（每项独立加分）**

- Props 按用途分组（如「状态控制」「交互事件」「样式定制」），非字母平铺排列 → +满分×7%
- 回调函数有参数展开说明（如 `onClick: function({ key, keyPath, domEvent })`），不只写类型名或 `Function` → +满分×7%
- 必填 prop 用 `*` 标注（不只依赖 TypeScript 的 `?` 省略） → +满分×4%
- 说明列有实质内容，不是「失效状态」「控制大小」这类等于没说的废话 → +满分×4%
- 类型文件规范（体现已建立 `types/index.ts` 统一类型导出）：API 表格类型列无 `any`；联合类型各值有注释说明 → +满分×3%

**扣分项（仅极端情形，上限 -5 分）**

- 超过半数属性说明列为空或只写了类型 → -3分
- 类型全部写 `any`，完全无参考价值 → -3分
- 同一组件的 props 和事件回调被按「属性 / 事件」强行拆成两张或多张表 → -3分

---

### 对接人（独立检查项，计入 PART1 冲刺分）

对接人不单独计分，但作为 PART1 冲刺项的评分依据。

**位置要求**：对接人必须以 `## 对接人` 作为独立二级标题，位置在 `## API` 之后、`## FAQ` 之前（或文档末尾，若无 FAQ）。

**格式要求**：每位对接人使用钉钉跳转链接格式：
`[{name}(前端)](dingtalk://dingtalkclient/action/sendmsg?dingtalk_id={dingdingId})`

⚠️ `{dingdingId}` 须手动替换为真实钉钉 ID，`{name}` 替换为真实姓名。若仍为占位符则视为未完成。

**对接人冲刺加分标准（计入 PART1）**：
- 有 `## 对接人` 独立模块，位置正确（API 后、FAQ 前），格式为钉钉跳转链接，且 `{dingdingId}` 已替换为真实值 → +PART1满分×5%（取整）

**以下情形不得此项分**：
- 对接人混写在「何时使用」或其他段落中
- 有 `## 对接人` 模块但格式为纯文本（如 `@张三`），无钉钉跳转链接
- 有链接格式但 `{dingdingId}` 仍是占位符未替换

**以下情形触发扣分**：
- 对接人写在「何时使用」模块内 → -1分（位置错误）

---

### PART 4：FAQ（仅在文档中存在时参与计分）

满分 24 分，保底 = 满分 × 75%（取整）= 18 分。
若文档不含 FAQ，此部分跳过，分值已分摊至其余部分，不影响总分和保底。

**保底标准（同时满足以下 2 条，得保底分）**

1. 有 FAQ 模块，且至少有 1 条问答（1 条即满足保底）
2. 该问题有对应的答案正文

**冲刺标准（每项独立加分）**

- 问题标题用「症状型」描述（如「Form 里点了没反应？」而非「valuePropName 的用法」） → +满分×10%
- 答案结构为「一句话结论 → ❌/✅ 对比代码 → 一句话根因」，先给答案再解释原因 → +满分×10%
- FAQ 按使用频率或上手顺序排列，高频问题在前（仅 2 条以上时才评此项） → +满分×5%

**扣分项（仅极端情形，上限 -3 分）**

- FAQ 链接到全局汇总页而非当前组件文档下 → -2分
- 所有问题标题都是 API 概念名，无一条症状型描述 → -2分
```
